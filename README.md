Практическое занятие 7: Базовые методы аутентификации
В рамках этой практики был реализован базовый сервер на Node.js с использованием Express. Основное внимание уделено хешированию паролей с помощью библиотеки bcrypt и созданию двух основных маршрутов аутентификации: регистрации (/api/auth/register) и входа (/api/auth/login), где пароль перед сохранением хешируется, а при входе сравнивается с хешем из базы данных (в текущей реализации — из массива в памяти).

Практическое занятие 8: Базовые методы аутентификации (JWT)
На этом этапе работа была расширена за счет внедрения JSON Web Tokens (JWT). При успешном входе в систему сервер теперь генерирует и возвращает access-токен. Также был добавлен защищенный маршрут /api/auth/me, доступ к которому контролируется через middleware authMiddleware, проверяющий наличие и валидность переданного токена в заголовке Authorization.

Практическое занятие 9: Refresh-токены
В данной работе была реализована более надежная схема аутентификации с использованием пары токенов. Помимо access-токена с коротким временем жизни (15 минут), был добавлен refresh-токен с длительным сроком действия (7 дней). Также был создан новый эндпоинт /api/auth/refresh, который принимает refresh-токен и, после его проверки, генерирует и возвращает новую пару токенов, обеспечивая бесшовное продление сессии пользователя.

Практическое занятие 10: Хранение токенов на фронтенде
Для этого задания была разработана фронтенд-часть приложения с использованием React. Реализованы страницы регистрации и входа, а также интерфейс для управления товарами (CRUD операции). Взаимодействие с сервером настроено через HTTP-клиент с перехватчиками (axios interceptors), которые автоматически добавляют access-токен в заголовки запросов и обрабатывают ошибки 401, отправляя запрос на обновление токена.

Практическое занятие 11: Управление доступом на основе ролей (RBAC)
В завершающей практике в приложение была добавлена система ролей (гость, пользователь, продавец, администратор). На сервере создан middleware roleMiddleware, который в связке с authMiddleware ограничивает доступ к определенным маршрутам в зависимости от роли пользователя. На фронтенде реализованы соответствующие права: например, продавцы могут создавать и редактировать товары, а администратор — управлять пользователями и удалять товары.
